# 모바일 UX 개선 (모바일 메뉴 검색 + 광고 슬롯 재배치 + 접근성)

> 상태: 기획 확정
> 확정일: 2026-06-12
> 다음 산출물: `docs/design/202606/mobile-ux-improvement-design.md` (구현 직전 설계 문서)

## 0. 확정 결정사항

| 항목 | 결정 |
| --- | --- |
| 작업 범위 | 모바일 화면(`max-width: 768px`)에 한정. 데스크톱 `.site-nav` 드롭다운은 변경 없음 |
| 핵심 목표 | (1) 모바일 메뉴 검색 기능 추가, (2) 메뉴 구조를 아코디언 + 인기 Top 6 형태로 재구성, (3) 모바일 광고 슬롯을 콘텐츠 사이로 명시 배치, (4) 모바일 메뉴 접근성 보강 |
| 데이터 소스 | 신규 데이터 파일 생성하지 않음. `src/data/tools.ts` + `src/data/reports.ts`를 그대로 검색/그룹핑 대상으로 사용 |
| 검색 구현 방식 | 클라이언트 사이드 vanilla JS. 빌드 시점에 `tools.ts`/`reports.ts`에서 검색용 JSON을 생성해 `<script type="application/json">` 또는 inline 배열로 주입 (외부 API 호출 없음) |
| 광고 정책 | 기존 Auto Ads는 유지. 계산기 입력 영역·모바일 메뉴 내부에는 광고가 절대 노출되지 않도록 `data-ad-layout`/CSS 격리. 신규 명시적 슬롯은 콘텐츠 섹션 사이(예: 계산 결과 ↔ FAQ 사이)에만 1차로 1곳 추가 |
| 접근성 기준 | WAI-ARIA 모바일 dialog 패턴 — ESC 닫기, 포커스 트랩, body 스크롤 잠금, 열릴 때 검색 input 자동 포커스 |
| 작업 분리 | 1차(검색+아코디언+Top6) → 2차(광고 슬롯) → 3차(접근성)로 순차 배포. 각 단계 독립적으로 커밋/배포 가능해야 함 |

### 0-1. 1차 MVP에 반드시 포함할 것

- 모바일 메뉴 상단 검색 입력창(`<input type="search">`)
- 검색어 입력 시 `tools.ts` + `reports.ts`의 `title`/`description`/`badges`/`category`를 대상으로 부분 일치 필터링
- 검색 결과를 "계산기" / "리포트" 두 그룹으로 분리해 표시 (그룹별 최대 6개, "더 보기" 링크로 전체 결과 페이지 이동)
- 검색어가 없을 때: 기존 4개 섹션(계산기/리포트/비교표/사이트)을 **아코디언(접기/펼치기)**으로 전환 + 최상단에 "인기 계산기 Top 6" 고정 섹션 추가
- 검색 입력 시 디바운스(150ms) 처리, 결과 0건일 때 안내 문구 표시

### 0-2. 1차 MVP에서 하지 않을 것

- 검색 자동완성/추천 키워드 드롭다운
- 검색 로그 수집/분석 연동 (GA 이벤트는 2차 이후 검토)
- 데스크톱 검색 UI (데스크톱은 드롭다운 메뉴로 충분히 탐색 가능하다고 판단)
- 광고 슬롯 신규 추가 (1차는 메뉴 구조 개선만, 광고는 2차)

### 0-3. 이후 단계

| 단계 | 내용 |
| --- | --- |
| 2차 | 모바일 콘텐츠 페이지에 명시적 `<ins class="adsbygoogle">` 슬롯 1곳 추가 (계산 결과와 FAQ/SEO 콘텐츠 사이), 계산기 입력 영역·모바일 메뉴에는 `data-ad-layout="no-ad"` CSS로 광고 차단 영역 명시 |
| 3차 | 모바일 메뉴 접근성: ESC 닫기, 포커스 트랩, body 스크롤 잠금, 검색 input 자동 포커스, `aria-live`로 검색 결과 개수 안내 |

## 2. 배경

- 네이버 서치어드바이저 데이터 분석 결과 모바일 트래픽의 클릭률(CTR)이 데스크톱보다 높게 나타남 → 모바일 경험이 사이트 전체 성과에 미치는 영향이 큼
- 현재 모바일 메뉴(`.site-header__mobile-menu`)는 계산기 17개, 리포트 11개, 비교표 5개, 사이트 3개 — 총 36개 링크가 **펼쳐진 플랫 리스트**로 한 화면에 모두 노출됨
  - 스크롤이 매우 길어지고, 사용자가 원하는 항목을 찾기 위해 전체를 훑어야 함
  - 카테고리 태그(`.site-header__mobile-cat`)가 붙어 있지만 시각적으로만 구분되고 그룹핑/접기 기능은 없음
- 검색 기능이 전혀 없어, "연봉"처럼 명확한 의도를 가진 사용자도 메뉴를 끝까지 스크롤해야 함
- 광고는 Auto Ads(`ca-pub-4445668013193986`)에 전적으로 의존 — 위치를 직접 제어할 수 없어 계산기 입력 영역 근처에 광고가 삽입될 위험이 있고, 모바일에서는 화면이 좁아 광고 밀도가 더 크게 느껴짐
- 모바일 메뉴 오버레이에는 ESC 닫기, 포커스 트랩, 스크롤 잠금 등 표준 다이얼로그 접근성 패턴이 없음

→ 위 세 가지(탐색성, 광고 배치, 접근성)를 한 번에 다루되, 리스크를 줄이기 위해 단계별로 분리 배포한다.

## 3. 모바일 메뉴 역할 정의

| 영역 | 변경 전 | 변경 후 |
| --- | --- | --- |
| 메뉴 최상단 | (없음) | 검색 입력창 (`role="search"`) |
| 검색 결과 없을 때 | 4개 섹션 풀 펼침 | "인기 계산기 Top 6" 고정 + 4개 섹션 아코디언(기본 접힘, 단 "계산기" 섹션은 기본 펼침) |
| 검색 결과 있을 때 | (해당 없음) | "계산기" 그룹 / "리포트" 그룹으로 분리된 결과 리스트, 그룹별 최대 6개 + "전체 결과 보기" 링크 |
| 광고 | Auto Ads만 존재, 위치 미지정 | 메뉴 영역은 `data-ad-layout="no-ad"`로 광고 제외, 본문 콘텐츠 사이 1개 슬롯(2차) |
| 닫기/포커스 | 햄버거 클릭, 오버레이 클릭만 | + ESC 키, 포커스 트랩, body 스크롤 잠금, 열릴 때 검색 input 자동 포커스(3차) |

## 4. 확정 IA (모바일 메뉴 구조)

```
[모바일 메뉴 오픈]
├─ 검색 입력창 (placeholder: "계산기·리포트 검색 (예: 연봉, 보유세)")
│
├─ [검색어 있음]
│   ├─ 계산기 결과 (최대 6개) — 없으면 섹션 숨김
│   ├─ 리포트 결과 (최대 6개) — 없으면 섹션 숨김
│   ├─ 결과 0건 → "검색 결과가 없습니다. '전체 계산기 보기'에서 찾아보세요"
│   └─ "전체 검색 결과 보기 →" (/tools/?q=검색어 또는 /search/)
│
└─ [검색어 없음]
    ├─ 인기 계산기 Top 6 (고정, 아코디언 아님)
    ├─ ▸ 계산기 (아코디언, 기본 펼침)
    ├─ ▸ 리포트 (아코디언, 기본 접힘)
    ├─ ▸ 비교표 (아코디언, 기본 접힘)
    ├─ ▸ 사이트 (아코디언, 기본 접힘)
    └─ "전체 계산기 보기 →" CTA
```

### 4-1. "인기 계산기 Top 6" 선정 기준

- `src/data/tools.ts`의 `badges`에 `"추천"`이 포함된 항목 우선
- 동률 시 `order` 값이 낮은(상위 노출) 항목 우선
- 최초 후보(현재 데이터 기준, 구현 시 재확인): 연봉 티어 계산기, 가구 소득 계산기, 한 명만 육아휴직, 적립식 투자 수익 비교, 파이어족 계산기, 국민연금 계산기
- 운영 중 교체가 쉬워야 하므로 **하드코딩 slug 배열**로 관리 (예: `MOBILE_POPULAR_SLUGS` in `tools.ts`)

## 5. 검색 기능 상세

### 5-1. 검색 대상 데이터

- `src/data/tools.ts`의 `ToolMeta[]` (slug, title, description, category, badges)
- `src/data/reports.ts`의 `ReportMeta[]` (slug, title, description, badges)
- 신규 데이터 파일을 만들지 않고, Astro 빌드 시점에 두 배열을 합쳐 검색용 경량 JSON(slug, title, type, href, category)을 생성해 `SiteHeader.astro`에 inline `<script type="application/json" id="mobile-search-data">`로 주입

### 5-2. 매칭 로직

- 입력값을 trim + 소문자 변환(한글은 대소문자 영향 없음) 후, `title` 또는 `description`에 부분 문자열로 포함되는지 검사
- 1차는 단순 `includes()` 매칭으로 충분 — 형태소 분석/오타 허용은 범위 밖
- 검색어 길이 1자 이상부터 동작, 디바운스 150ms

### 5-3. 결과 렌더링

- 계산기 결과: `.site-header__mobile-cat`에 `category` 표시, 최대 6개
- 리포트 결과: `.site-header__mobile-badge`로 "리포트" 라벨 표시(선택), 최대 6개
- "연봉" 검색 예시 → 계산기 그룹: 연봉 티어 계산기, 가구 소득 계산기, 연봉 인상 계산기, 부부 연봉 순위 계산기 등 / 리포트 그룹: 2026 신입사원 초봉 비교, IT 업계 신입 초봉 TOP10, 연봉·자산 2016 vs 2026 등

## 6. 광고 슬롯 전략 (2차)

| 항목 | 내용 |
| --- | --- |
| 현재 상태 | Auto Ads만 적용 (`BaseLayout.astro` 65-66행), 명시적 `<ins class="adsbygoogle">` 없음 |
| 문제 | Auto Ads는 위치를 구글이 자동 결정 → 계산기 입력 영역 직전/직후, 모바일 메뉴 내부 등 부적절한 위치에 삽입될 가능성 |
| 1차 조치 | 계산기 입력 영역(`SimpleToolShell`/`CompareToolShell`/`TimelineToolShell`의 input 컨테이너)과 모바일 메뉴(`.site-header__mobile-menu`)에 `data-ad-layout="no-ad"` 속성 부여 + Auto Ads 제외 설정 검토 |
| 2차 조치 | 콘텐츠 섹션 사이(계산 결과 카드 ↔ `SeoContent`/FAQ 사이)에 명시적 `<ins class="adsbygoogle">` 슬롯 1개 추가. 모바일 전용 슬롯 크기 사용 |
| 측정 | 배포 후 AdSense 리포트에서 모바일 viewability/CTR 변화 1주간 관찰 |

> 본 계획서는 광고 슬롯의 "위치 원칙"만 확정한다. 실제 슬롯 코드/스타일은 2차 착수 시 `docs/design/`에 별도 설계 문서로 정리한다.

## 7. 접근성 개선 (3차)

| 항목 | 현재 | 개선 |
| --- | --- | --- |
| ESC 닫기 | 없음 | `keydown` 리스너로 `Escape` 시 메뉴 닫기 |
| 포커스 트랩 | 없음 | 메뉴 열림 상태에서 `Tab`/`Shift+Tab`이 메뉴 내부 요소 사이에서만 순환 |
| 스크롤 잠금 | 없음 | 메뉴 열릴 때 `<body>`에 `overflow: hidden` 적용, 닫힐 때 해제 |
| 초기 포커스 | 없음 | 메뉴 열릴 때 검색 input에 자동 포커스 |
| 결과 안내 | 없음 | 검색 결과 영역에 `aria-live="polite"`로 "N개 결과" 안내 |
| 닫을 때 포커스 복귀 | 없음 | 닫힐 때 포커스를 햄버거 버튼으로 복귀 |

## 8. 화면 구성 가이드

- 검색 입력창: 모바일 메뉴 최상단, sticky 없이 메뉴 스크롤 영역(`.site-header__mobile-inner`) 안쪽 최상단에 고정 배치(스크롤 시 함께 스크롤 — 1차에서는 sticky 미적용, 필요 시 추후 검토)
- 아코디언 헤더: 기존 `.site-header__mobile-label`을 클릭 가능한 `<button>`으로 변경, `aria-expanded` + `▾`/`▴` 아이콘 토글
- 인기 계산기 Top 6: 아코디언이 아닌 고정 섹션, 검색 입력창 바로 아래 배치
- 검색 결과 그룹 헤더: "계산기 (N)" / "리포트 (N)" 형태로 개수 표시
- 색상/타이포: 기존 디자인 토큰(`_tokens.scss`) 그대로 사용, 신규 색상 추가 없음

## 9. SEO 전략

- 모바일 메뉴 자체는 검색엔진 색인 대상이 아니므로 SEO에 직접 영향 없음
- 다만 메뉴 탐색성 개선으로 세션당 페이지뷰가 늘어나면 사이트 체류시간/이탈률 지표 개선 → 간접적으로 SEO 신호에 긍정적
- "전체 검색 결과 보기" 링크가 `/tools/` 라이브러리 페이지로 연결되도록 하여, 기존 `/tools/` 페이지의 내부 링크 수도 함께 증가

## 10. 구현 우선순위

### 1차 (이번 스프린트)

1. `tools.ts`/`reports.ts` 기반 검색 데이터 생성 (Astro frontmatter에서 inline JSON 또는 `<script type="application/json">`)
2. `SiteHeader.astro` 모바일 메뉴 상단에 검색 input 추가
3. 검색어 입력 시 계산기/리포트 그룹별 결과 렌더링 (vanilla JS, `<script>` 블록)
4. 검색어 없을 때: "인기 계산기 Top 6" 고정 섹션 + 기존 4개 섹션 아코디언화
5. `_legacy.scss`에 검색 input, 아코디언 토글, Top 6 섹션 스타일 추가
6. `npm run build`로 전체 페이지 정상 빌드 확인 + 모바일 뷰포트 수동 QA

### 2차

1. 계산기 입력 영역 / 모바일 메뉴에 `data-ad-layout="no-ad"` 부여 및 Auto Ads 제외 검토

3. 배포 후 AdSense 모바일 지표 1주 관찰

## 12. 결론

모바일 트래픽의 비중과 CTR이 높은 상황에서, 현재의 평면적인 36개 링크 모바일 메뉴는 탐색 비용이 크고 광고 배치도 제어되지 않으며 접근성 기준도 충족하지 못한다. 이를 (1) 검색+아코디언+인기 Top6 메뉴 개편, (2) 콘텐츠 사이 명시적 광고 슬롯 1곳 도입, (3) 표준 모바일 다이얼로그 접근성 패턴 적용의 3단계로 분리해 순차 배포한다. 각 단계는 독립적으로 검증·배포 가능하며, 1차는 신규 데이터 파일 없이 기존 `tools.ts`/`reports.ts`만으로 구현 가능해 리스크가 낮다. 1차 완료 후 실제 모바일 사용성 변화를 확인한 뒤 2차(광고)·3차(접근성)로 진행한다.
