## 1. 왜 새로 만들까

네이버 검색어 `생계급여 소득인정액 계산기`가 적은 노출에서도 높은 CTR을 보였다. 현재 비교계산소에는 이미 `/tools/welfare-benefit-eligibility/`가 있고, 이 페이지는 생계·의료·주거·교육급여를 모두 보는 통합 계산기다.

따라서 이 기획은 완전히 다른 계산 로직을 새로 만드는 것이 아니라, 기존 복지급여 계산 엔진을 재사용하되 **생계급여만 빠르게 계산하고 예상 지급액까지 강조하는 검색 의도 전용 페이지**를 추가하는 방향이다.

핵심 차이는 다음과 같다.

| 구분 | 기존 복지급여 자격 계산기 | 신규 생계급여 소득인정액 계산기 |
|---|---|---|
| URL | `/tools/welfare-benefit-eligibility/` | `/tools/livelihood-benefit-income-recognition/` |
| 핵심 의도 | 기초생활보장 4대 급여 가능성 전체 확인 | 생계급여 가능 여부와 예상액 빠른 확인 |
| 검색어 | 기초생활수급자 자격 계산기, 복지급여 자격 | 생계급여 소득인정액 계산기, 생계급여 얼마 |
| 첫 결과 | 소득인정액 + 급여별 가능성 | 생계급여 기준선, 예상 생계급여액, 탈락까지 차이 |
| UX | 입력 항목이 넓고 종합적 | 빠른 입력, 생계급여 중심 결과, 상세 항목 접기 |
| 내부 역할 | 모 계산기 | 검색 유입용 하위 계산기/랜딩 |

## 7. 계산 로직

### 기본 구조

```text
소득인정액 = 소득평가액 + 재산의 소득환산액
예상 생계급여액 = max(생계급여 선정기준 - 소득인정액, 0)
생계급여 기준 대비 비율 = 소득인정액 / 생계급여 선정기준
```

### 소득평가액

```text
월소득합계 = 근로·사업소득 + 공적이전소득 + 사적이전소득 + 재산소득
근로소득공제 = min(근로·사업소득 × 30%, 600,000원)
소득평가액 = max(월소득합계 - 근로소득공제, 0)
```

공제율은 기존 계산기와 같은 간이 모델을 쓴다. 실제 제도는 가구 특성별 공제 항목이 다를 수 있으므로 결과에 `추정` 배지를 표시한다.

### 재산의 소득환산액

```text
순재산 = max(주거재산 + 일반재산 + 금융재산 - 기본재산액 - 부채, 0)
재산의 소득환산액 = 주거·일반재산 환산액 + 금융재산 환산액
```

MVP에서는 기존 계산기와 동일하게 다음 원칙을 적용한다.

- 대도시, 중소도시, 농어촌별 기본재산액 차감
- 금융재산은 별도 환산율 적용
- 자동차는 금액을 바로 반영하기보다 `확인 필요` 위험 플래그로 표시

### 판정 단계

| 단계 | 조건 | 메시지 |
|---|---|---|
| 가능성 높음 | 소득인정액 <= 선정기준 95% | 생계급여 신청 가능성이 있는 구간입니다. |
| 경계 | 선정기준 95% 초과~105% 이하 | 입력값과 공제 반영에 따라 결과가 달라질 수 있습니다. |
| 초과 | 선정기준 105% 초과 | 생계급여 기준은 초과한 것으로 보입니다. |
| 확인 필요 | 자동차·부양의무자·재산 특이값 존재 | 주민센터 상담이 필요한 항목이 있습니다. |

## 12. 결론

이 페이지는 기존 복지급여 통합 계산기와 경쟁하면 안 된다. 역할은 명확하다. 사용자가 `생계급여 소득인정액 계산기`처럼 매우 구체적인 검색어로 들어왔을 때, 첫 화면에서 바로 **생계급여 기준선과 예상 지급액**을 보여주는 전용 입구가 되어야 한다.

구현은 신규 계산 엔진보다 **기존 엔진 재사용 + 결과 화면 재구성 + 정확매칭 SEO**가 핵심이다.
