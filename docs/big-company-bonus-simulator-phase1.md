# 대기업 성과급 시뮬레이터 Phase 1 분석/설계

## 1. 이 사이트에 맞는 포지셔닝

현재 저장소는 `블로그 본문에 링크 또는 iframe으로 붙이는 독립형 도구 페이지` 구조다.
따라서 이 기획은 `시리즈형 콘텐츠 묶음`보다 먼저 `대표 계산기 1개`로 들어가는 편이 맞다.

가장 적절한 1차 배치는 아래와 같다.

- 메인 도구 페이지 1개: `/tools/bonus-simulator/`
- 블로그 본문 연계 방식:
  - 메인 블로그 글에서 계산기 링크
  - 필요 시 iframe 삽입
  - 회사별 해설 글은 본 블로그에서 별도 운영

정리하면, 이 저장소 안에서는 `정적 인터랙티브 계산기`가 주인공이고, `시리즈형 설명 글`은 외부 블로그 본문이 맡는 구조가 가장 자연스럽다.

## 2. 추천 정보 구조

### 2-1. Phase 1 권장 페이지 구성

Phase 1은 한 페이지 완결형이 적합하다.

- 페이지명: `대기업 성과급 시뮬레이터`
- 슬러그 후보:
  - `bonus-simulator`
  - `big-company-bonus`
  - `performance-bonus`

권장안은 `bonus-simulator`다.
짧고, 현재 `salary`, `retirement`, `negotiation` 같은 기존 슬러그 스타일과도 잘 맞는다.

### 2-2. 페이지 내 섹션 구조

1. Hero
2. 빠른 입력 카드
3. 결과 요약 카드
4. 올해 성과급 분해
5. 3개년 총보상 추이
6. 동일 직급 3사 비교
7. 해석 카드
8. 근거/면책 안내

이 구성은 현재 사이트의 `Hero -> Input -> Summary -> Table/Notice` 패턴을 유지하면서도, 이번 기획의 핵심인 `비교`와 `3개년 시뮬레이션`을 확장하기 쉽다.

## 3. 왜 단일 계산기부터 가야 하는가

현재 코드베이스는 아래 성격이 강하다.

- `tools.ts` 기반의 평면적인 도구 목록
- `/tools/<slug>/` 단위의 독립 페이지
- `public/scripts/tools.js` 한 파일에서 여러 계산기 로직 처리
- 공통 카드/표/안내 컴포넌트 중심

이 구조에서 처음부터 아래를 동시에 넣으면 과하다.

- 삼성전자 전용 페이지
- SK하이닉스 전용 페이지
- 현대자동차 전용 페이지
- 비교 전용 페이지
- 시리즈 랜딩 페이지

이렇게 가면 도구 개수는 늘지만 사용자는 어느 페이지부터 봐야 하는지 헷갈린다.
반대로 Phase 1은 `회사 선택 -> 직급 선택 -> 결과 확인` 한 페이지로 끝내야 현재 사이트 톤과 맞다.

## 4. 사용자 가치 기준 핵심 설계

사용자가 이 도구에서 얻고 싶은 것은 정밀 HR 계산이 아니라 아래 4가지다.

- 올해 성과급 감각
- 같은 직급 기준 회사 간 비교
- 업황/시나리오에 따른 차이
- 3년 누적 보상 그림

따라서 입력은 적게, 결과 해석은 강하게 가야 한다.

권장 최소 입력은 아래 4개다.

- 회사
- 직급
- 연봉
- 시나리오

상세 입력은 접힘 영역으로 내린다.

- 월 기본급/통상임금
- 성장 기대치
- 현대차 주식 반영 여부

## 5. 이 사이트 기준 UX 설계안

### 5-1. 첫 화면

첫 화면 안에서 계산 완료가 가능해야 한다.

- 회사 선택
- 직급 선택
- 연봉 자동 추천
- 시나리오 선택
- 계산 버튼

현재 사이트 스타일상 좌측 입력 패널, 우측 결과 영역의 2열 구성이 잘 맞는다.
모바일에서는 1열로 자연스럽게 내려가면 된다.

### 5-2. 결과 우선순위

상단 KPI 카드 4개를 유지하는 것이 좋다.

- 올해 예상 성과급
- 올해 예상 총보상
- 2026~2028 3년 누적 총보상
- 회사 내 체감 포지션

이후 상세 영역에서 아래를 보여준다.

- 성과급 구성 분해
- 3개년 추이
- 3사 비교
- 한 줄 해석

### 5-3. 표현 방식

현재 프로젝트는 표 중심 결과가 강하다.
이번 계산기는 표만으로는 재미가 약하므로, 아래 방식으로 바꾸는 편이 낫다.

- 핵심 수치: 카드
- 올해 3사 비교: 가로 막대형 비교 블록
- 2026~2028: 간단한 추이 그래프 또는 연도별 카드
- 세부 근거: 표 또는 리스트

즉, `표는 보조`, `카드는 메인`이 더 적합하다.

## 8. 계산 로직 설계 원칙

### 8-1. 현재 프로젝트에 맞는 로직 방향

이 프로젝트는 프론트 정적 계산이 핵심이므로, 계산식은 `설정값 + 시나리오 계수` 중심이어야 한다.

권장 원칙:

- 회사별 설정값은 TS 상수로 분리
- 계산 로직은 회사 공통 함수 + 회사별 분기 최소화
- 사용자가 직접 숫자 수정 가능
- 결과는 `정확한 실지급`이 아니라 `예상 범위/시나리오`로 표현

### 8-2. 회사별 단순 모델

삼성전자:

- `성과급 = TAI + OPI`
- 조직 프리셋은 3단계
  - 보수적 조직
  - 평균 조직
  - 상위 조직

SK하이닉스:

- `성과급 = PI + PS`
- 업황 프리셋 3단계
  - 다운사이클
  - 노멀
  - 슈퍼사이클

현대자동차:

- `성과급 = 월 기준급 x 성과급 비율 + 정액 성과금`
- 추가 보상:
  - 주식
  - 상품권

### 8-3. 결과 표현 원칙

정확 수치처럼 단정하지 말고 아래 표현을 유지한다.

- `예상 성과급`
- `시나리오 기반 총보상`
- `공개 자료를 바탕으로 단순화한 추정`

## 9. 데이터 구조 권장안

현재 `public/scripts/tools.js` 하나에 모든 계산기를 넣는 구조는 작은 도구에는 괜찮지만, 이번 기획부터는 복잡도가 커진다.
따라서 이번 도구는 아래 방향으로 분리하는 것이 좋다.

- `src/data/bonusSimulator.ts`: 회사별 설정값
- `public/scripts/bonus-simulator.js`: 페이지 전용 계산 로직
- `src/pages/tools/bonus-simulator.astro`: 페이지

추천 타입 초안:

```ts
export type CompanyCode = "SAMSUNG" | "SKHYNIX" | "HYUNDAI";
export type Rank = "STAFF" | "ASSISTANT_MANAGER" | "MANAGER" | "DEPUTY_GM" | "GM";
export type Scenario = "CONSERVATIVE" | "BASE" | "AGGRESSIVE";

export interface BonusPreset {
  label: string;
  bonusMultiplier: Record<Scenario, number>;
  description: string;
}

export interface FutureProjection {
  salaryGrowth: Record<Scenario, number>;
  bonusGrowth: Record<Scenario, number>;
}

export interface CompanyConfig {
  companyCode: CompanyCode;
  companyName: string;
  heroLabel: string;
  bonusModel: "SAMSUNG_TAI_OPI" | "SK_PI_PS" | "HYUNDAI_PACKAGE";
  baseSalaryByRank: Record<Rank, number>;
  presets: BonusPreset[];
  futureProjection: FutureProjection;
  notes: string[];
}
```

이 구조면 DB 없이도 유지보수가 쉽고, 나중에 JSON/API로 옮기기도 쉽다.

## 12. 시리즈 관점에서의 확장 전략

이 저장소 안의 시리즈는 `도구 시리즈`로 해석하는 편이 맞다.
즉, 한 번에 글 여러 편을 만드는 것이 아니라 아래 순서가 적절하다.

1. `bonus-simulator` 대표 페이지 출시
2. 반응이 좋으면 회사별 파생 계산기 검토
3. 이후 메인 블로그에서 해설형 글 시리즈 전개

파생 후보:

- `/tools/bonus-simulator/` 메인 비교형
- `/tools/sk-bonus/` 업황 민감도 특화형
- `/tools/hyundai-bonus/` 패키지형 보상 특화형

하지만 Phase 1에서는 메인 비교형 하나면 충분하다.

## 13. SEO/블로그 연계 관점

이 프로젝트 자체는 툴 저장소라서 SEO 허브보다는 `도구 랜딩` 역할이 강하다.
따라서 검색형 제목을 일부 반영하되, 페이지 목적은 계산 완료에 둔다.

권장 메타 방향:

- title: `대기업 성과급 시뮬레이터 | 삼성전자·SK하이닉스·현대차 비교`
- description: `삼성전자, SK하이닉스, 현대자동차의 직급별 성과급과 2026~2028 총보상 시나리오를 빠르게 비교하는 정적 계산기`

블로그 글과의 연결 문구 예시:

- `삼성전자 OPI/TAI 구조를 먼저 읽고 계산기로 비교해보세요.`
- `하이닉스 슈퍼사이클 가정을 넣으면 3년 누적 차이를 바로 확인할 수 있습니다.`

- `실제 지급액 보장`
- `회사 공식 기준`
- `정확 계산`

## 15. 최종 권장안

현재 사이트에 가장 잘 맞는 Phase 1 해법은 아래다.

- 이 저장소에는 `대기업 성과급 시뮬레이터` 단일 대표 페이지를 추가한다.
- 슬러그는 `/tools/bonus-simulator/`를 권장한다.
- 한 페이지 안에서 `입력 -> 올해 결과 -> 3년 추이 -> 3사 비교 -> 안내`까지 끝낸다.
- 회사별 긴 해설 콘텐츠는 이 저장소가 아니라 메인 블로그 글에서 담당한다.
- 구현 시에는 기존 공통 컴포넌트를 재사용하되, 데이터와 스크립트는 이번 도구부터 분리한다.

이 방향이면 현재 프로젝트의 `독립형 정적 툴` 성격을 해치지 않으면서도, 사용자가 원하는 `재미 + 납득 + 빠른 비교`를 모두 살릴 수 있다.
